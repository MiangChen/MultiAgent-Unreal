# 兼容性设计记录（Gantt / SceneGraph / Runtime）

更新时间：2026-02-28

## 背景

当前 Gantt 拖拽架构已按目标收敛为：

- `Canvas -> DragStateMachine`
- `DragStateMachine -> (DragSource / DropTarget / Preview)`
- `Canvas -> Painter`

中间层 `DragController` 已移除，避免与目标架构不一致。

## 当前保留的上层兼容接口

为了不破坏 `Viewer / Modal / Blueprint` 的既有调用，`UMAGanttCanvas` 保留了以下公开接口与事件：

- `IsDragging()`
- `SetDragEnabled(bool)`
- `IsDragEnabled()`
- `CancelDrag()`
- `OnDragStarted`
- `OnDragCompleted`
- `OnDragFailed`
- `OnDragBlocked`
- `OnSkillDragCancelled`

说明：这些接口属于“上层产品语义”，不是内部实现细节。内部实现可继续演进，但这些入口暂时保持稳定。

## 未来可删除/收敛的兼容项

以下项可以在满足条件后删除或收敛：

- `UMAGanttCanvas` 中仅用于历史调用习惯的公开拖拽方法（如未来改为统一交互门面时）。
- 仅用于向旧调用方转发事件的绑定代码（如果上层统一改为新事件总线）。

## 删除前检查清单

执行删除前，至少满足：

- `MASkillAllocationViewer` 不再直接依赖这些旧入口。
- `MASkillAllocationModal` 不再直接依赖这些旧入口。
- 相关 Blueprint 引用已迁移（无断链）。
- 通过一次完整回归：编译、启动 `SetupMap`、拖拽交互（开始/完成/失败/取消/阻止）。

## 备注

如果未来确认不再需要上层兼容，请优先“分阶段删除”，先标记弃用，再移除实现，避免一次性破坏调用链。

## MANavigationService 兼容性记录（2026-02-27）

以下内容属于“兼容与兜底分支”，用于保证旧行为与异常场景可运行。未来可按条件逐步删除。

### 1) API 兼容层（暂保留）

- 公开接口保持不变：`NavigateTo / TakeOff / Land / ReturnHome / FollowActor / PauseNavigation / ResumeNavigation / CancelNavigation`
- 目的：避免上层技能、蓝图、行为树调用断裂。

### 2) 地面导航兼容兜底（暂保留）

- 无 `AIController` 时：回退到 `StartManualNavigation()`。
- NavMesh 失败、假成功、`AlreadyAtGoal` 但距离异常时：回退到手动导航。
- 目的：保证没有完整 AI/NAV 环境时也能继续导航。

### 3) 飞行导航兼容兜底（暂保留）

- `FlightController` 不可用时：回退到 `MOVE_Flying + 直线导航`。
- 飞行跟随中控制器不可用时：回退到简单直线跟随。
- 目的：避免飞控初始化失败直接中断核心流程。

### 4) Pause/Resume 兼容兜底（暂保留）

- `ResumeFromPauseSnapshot()` 中，如果原定时器不可恢复：
- 跟随模式重启 `FollowActor(...)`
- 手动导航重启 `StartManualNavigation()`
- 地面导航重启 `NavigateTo(...)`
- 目的：防止暂停恢复时因上下文不完整导致“恢复失败”。

### 可删除条件（建议全部满足后再删）

- 上层调用统一迁移完成（C++/Blueprint/技能脚本不再依赖旧入口语义）。
- 目标运行环境稳定（`AIController`、NavMesh、FlightController 始终可用）。
- 已有明确替代路径，并通过回归：`Navigate / Follow / ReturnHome / Pause-Resume / Cancel`。
- 对应 fallback 监控日志在真实场景中长期不触发。

### 关联文档

- 详细行为守卫与排障入口：`docs/MANavigationService_refactor_notes.md`

## SceneGraph 兼容性记录（2026-02-28）

以下兼容策略用于保证 SceneGraph 内核重构（Hex 化）过程中，不破坏现有 UI/Agent/Core 调用。

### 1) Facade API 兼容层（暂保留）

- `UMASceneGraphManager` 对外接口保持不变（查询、编辑、保存、动态更新等入口不改签名）。
- 上层继续按原方式调用 `UMASceneGraphManager`，不感知内部改造。

### 2) 内核解耦兼容策略（已落地）

- Manager 内部改为 `Port -> Service -> UseCase -> Adapter`，但外部调用链不变。
- 查询路径已切换到 QueryUseCases，写路径已切换到 CommandUseCases。
- 事件发布改为 `EMASceneChangeType` 强类型链路，但仍通过 manager 统一出口。

### 3) 保存策略兼容（RunMode）

- 保持原有语义：运行模式不允许写盘时，仍跳过落盘并保留内存工作副本。
- 目的：避免编辑/运行模式切换时出现行为回归。

### 可删除条件（建议全部满足后再删）

- 上层模块直接依赖 `UMASceneGraphManager` 旧入口的调用点完成迁移（如果计划升级到更细粒度门面）。
- 新接口具备同等可观测性（日志/事件/错误码）并通过完整回归。

## Runtime 兼容性记录（P5，2026-02-28）

以下内容属于“运行时稳定性兼容修复”，目标是清理噪音而不改变业务流程。

### 1) GameplayTag 兼容补齐

- 补充 `Skill.TakePhoto / Skill.Broadcast / Skill.HandleHazard / Skill.Guide` 到 `DefaultGameplayTags.ini`。
- 目的：兼容现有 `MASkillTags` 请求链路，消除 missing-tag ensure。

### 2) USTRUCT 初始化测试兼容

- 对运行时动态字段增加 `IgnoreForMemberInitializationTest`：
- `FMASceneChangeMessage::MessageId`
- `FMASkillAllocationMessage::MessageId`
- `FMAReviewResponseMessage::MessageId`
- `FMADecisionResponseMessage::MessageId`
- `FMAPIPCameraInstance::CameraId`
- 目的：保留现有动态生成逻辑，同时消除 UHT 成员初始化测试噪音。

### 3) TaskPlanner Mock 数据路径兼容

- 同步 `response_example.json` 到 `unreal_project/datasets/response_example.json`。
- 目的：兼容 `UMATaskPlannerWidget::LoadMockData()` 的既有读取路径。

### 4) StateTree 运行模式兼容（`use_state_tree=false`）

- 角色构造中统一 `SetStartLogicAutomatically(false)`。
- `MAStateTreeComponent` 在无资产时不自动启动；资产注入后再 `StartLogic()`。
- 目的：在禁用 StateTree 模式下，不触发“State Tree asset is not set”错误。

### 5) 导航噪音日志级别兼容

- `MANavigationService` 的 `FALSE SUCCESS` 诊断日志统一为 `Warning`。
- 目的：保留诊断信息，不把“可回退场景”误判为运行错误。
